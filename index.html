深入理解BFC

发表于 2022-09-01 分类于 CSS

BFC的定义

​ BFC全称为Block Formatting Context，直译为”块级格式化上下文“。它是一个独立的渲染区域。只有Block-level box参与，它规定了内部的Block-level box如何布局，并且与这个区域外部毫不相干。

block-level box即块级元素

BFC的生成

在MDN中，有总结BFC的生成方法：

  • 根元素（）

  • 浮动元素（元素的 float 不是 none）

  • 绝对定位元素（元素的 position 为 absolute 或 fixed）

  • 行内块元素（元素的 display 为 inline-block）

  • 表格单元格（元素的 display 为 table-cell，HTML表格单元格默认为该值），表格标题（元素的 display 为 table-caption，HTML表格标题默认为该值）

  • 匿名表格单元格元素（元素的 display 为 table、table-row、 table-row-group、table-header-group、table-footer-group（分别是HTML table、
    row、tbody、thead、tfoot 的默认属性）或 inline-table）

  • overflow 计算值(Computed)不为 visible 的块元素

  • 弹性元素（display 为 flex 或 inline-flex 元素的直接子元素）

  • 网格元素（display 为 grid 或 inline-grid 元素的直接子元素）

  • display 值为 flow-root 的元素

BFC的约束规则

  1. 在BFC中，box会在垂直方向上一个挨着一个排布；
  2. 垂直方向上的间距由margin属性决定；
  3. 在同一个BFC中，相邻两个box之间的margin会被折叠
  4. 在BFC中，每个元素的左边缘是紧挨着包含块的左边缘

BFC的作用

清除外边距margin折叠问题

​ 首先我们看个案例：

​ 我们分别给两个box设置了对应的margin，理论上两个box的间距应该是box1的margin-bottom的100px加上box2的margin-top的50px，一共为150px，可实际上由于这两个box处在同一个BFC中，造成了margin折叠，所以两个盒子之间的margin为较大的那个值，即100px。要解决margin折叠，我们可以让两个盒子处在两个BFC中，这样就可以解决margin折叠问题。

​ 我们可以给box2添加一个wrap，然后通过overflow:hidden触发这个box2的BFC，让box1和box2处于两个不同的BFC中，这样就可以解决外边距margin的折叠问题。

​ 可能有人会问，为啥不直接给box2设置overflow:hidden属性，这样box1和box2不也是处在两个不同的BFC了吗？答案肯定是不行的，前面我们说过，html根元素本身就会触发BFC，如果我们单纯的给box2设置，box1和box2依旧处在同一个BFC（根元素触发的BFC）中，而我们嵌套了一层后，box2处于的BFC是wrap创造的，box1处于的BFC是html根元素创造的。

函数模块化

发表于 2022-08-29 更新于 2022-08-30 分类于 JavaScript

什么是模块化

​ 首先来看维基百科对模块化的定义：

​ 模块化编程（英语：modular programming），是强调将计算机程序的功能分离成独立的、可相互改变的模块（module）的软件设计软件设计)技术，它使得每个模块都包含着执行预期功能的一个唯一方面（aspect）所必需的所有东西。

​ 通俗易懂的来讲，我们所说的模块化开发，就是将一个程序划分成一个个小的结构，按照这个结构划分去开发程序就是所谓的模块化开发。

​ 在每个结构结构中，我们可以编写对应的逻辑代码，而且这些逻辑代码在这个结构中有独有的作用域， 在这个结构可以将自己希望暴露的变量、函数、对象等导出给其结构使用；也可以通过某种方式，导入另外结构中的变量、函数、对象等。

模块化的历史背景

​ 摘自MDN：

​ JavaScript 程序本来很小——在早期，它们大多被用来执行独立的脚本任务，在你的 web 页面需要的地方提供一定交互，所以一般不需要多大的脚本。过了几年，我们现在有了运行大量 JavaScript 脚本的复杂程序，还有一些被用在其他环境（例如 Node.js）。

​ 因此，近年来，有必要开始考虑提供一种将 JavaScript 程序拆分为可按需导入的单独模块的机制。Node.js 已经提供这个能力很长时间了，还有很多的 JavaScript 库和框架已经开始了模块的使用（例如，CommonJS 和基于 AMD 的其他模块系统 如 RequireJS， 以及最新的 Webpack 和 Babel）。

​ 所以由此我们可以看出，无论你多么喜欢JavaScript这门语言，没有模块化是早期存在的极大痛点之一，虽然社区中给出了很多方案，但是JavaScript本身，直到ES6（2015）才推出了自己的模块化方案。而在此之前，为了让JavaScript支持模块化，涌现出了很多不同的模块化规范：AMD、CMD、CommonJS等

早期没有模块化的问题及解决方案

​ 没有模块化最大的问题之一就是命名冲突。

​ 假如甲乙在编写同一个程序，然后他们在各自实现对应的子功能的js文件里使用相同的变量名，最后在合并狼个人的代码时就会出现命名冲突的问题。

​ 在早期的解决方案中，我们可以用**IIFE(立即执行函数)**来解决，将逻辑代码用立即执行函数包裹成一个模块，将这个模块进行导出

​ 但是该方案看起来显然不优雅，首先使用立即执行函数包裹代码会显得代码看起来混乱，同时没有合适的规则情况下，模块的命名也会造成混乱。

AMD规范

​ AMD是Asynchronous Module Definition（异步模块定义）的缩写，它采用的是异步加载模块，而实现AMD规范我们常用的库是require.js。

​ 在main.js中配置需要导入的模块，然后在各个模块中，通过define这个函数来定义对应的模块

图片懒加载的实现方案

所谓的图片懒加载，就是在滑动网页的时候，碰到需要看到的图片才会去加载图片。

目的是为了减少网络请求，提高网页的加载速度。

那么如何控制图片的懒加载呢？

我们把这个问题拆分成两个部分：

  • 判断图片在什么时候出现
  • 当图片需要出现时，如何对它进行懒加载

这样就引出了我们以下的几种方案

方案一.图片的位置计算 + 滚动事件 + DataSet API

这个方案需要用到offsetTop、offsetParrent、clientHeight以及scrollTop

这几个概念我自己经常忘记或者搞混，所以说这个方案不是最佳的哈哈哈，但还是先简单分辨一下这几个概念。

可以从上面的图片看出，我们实现的逻辑是当图片的 offsetTop <= clientHeight + scrollTop 时，我们需要将图片进行加载，所以在图片还不需要加载的时候，我们可以给图片设置 data-src 属性，当需要加载的时候将 data-src 属性替换成 src 属性即可。

方案二.getBoundingClientRect API 的使用

这个方案就比较友好，我们可以通过获取图片的 DOM 节点，然后调用这个 api 直接获得图片距离网页顶部的高度 top，当 clientHeight >= top 时，说明图片已经出现在可视页面中，这时候我们就需要加载图片

方案三.图片的属性 lazyload

img 本身有个属性是 loading，我们可以把它设置 lazy 实现图片的懒加载，但是这个属性本身兼容性不是很好，所以用起来还是挺不爽的。

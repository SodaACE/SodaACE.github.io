useEffect

作用

​ 在该 Effect Hook 中，我们可以完成类似于 class 组件中生命周期的功能。类似于更新 DOM、发送网络请求这些操作，都是属于 React 中更新 DOM 的副作用，而操作这些副作用的函数，我们通常将其放入 Effect Hook中。

案例

​ 需求：我们定义一个加法计数器，当点击按钮时，计数值加 1，同时浏览器的 title 值会随着计数值的改变而改变。首先我们看不使用 hook 的版本

​ 我们需要在 componentDidMount 以及 componentDidUpdate 两个生命周期中操作我们的 DOM 才能完成需求，这看起来显然比较复杂，如果我们在多个页面中都需要使用类似的需求，我们都要在其组件的两个生命周期中编写相同的逻辑，这显然让代码看起来十分不优雅。

​ 在 class 组件中实现该需求看起来略显复杂，而函数式组件中的代码实现相同的需求看起来比较简单合理。

​ 在函数组件中，我们只需要一句话便可以完成上述需求，这看起来合理，但是该操作属于函数的副作用，在编写 React 代码中，我们应该尽量规避编类似的代码。

​ 结合上述两个案例，再来讲解 Effect Hook 会方便理解很多。

性能优化

​ 比如我们在 useEffect 中发送网络请求，同时组件中有一个计数器，当我们点击按钮增加计数值的时候，由于 render 会重新执行，所以整个组件又重新被渲染，然后执行组件中的代码，这样网络请求会再次发送造成性能浪费。

​ 而 useEffect 第二个参数就是解决类似的问题，它接受的是一个数组，里面的参数代表第一个参数函数中所依赖的项，只有当依赖项发生改变的时候，该函数才会重新执行。

​ 我们在 useEffect 的第二个参数写上一个空数组，代表第一个函数不依赖任何项，则只会在组件被创建的时候执行一次。

useCallback

作用

​ useCallBack 会返回一个函数的记忆值，在依赖不变的时候，多次定义某个函数，返回的值是相同的。通常我们使用这个 hook 是为了防止子组件多次渲染。

案例

​ 当我们使用普通函数版本的 increment，当我们修改 message 的时候，Home 组件会被重新渲染，这时候因为修改 message 的时候，组件被重新渲染，increment 被重新定义，而 increment 作为 props 传给 home 组件，当 props 中的属性发生改变时, 组件本身才会被重新渲染，所以 home 会重新被渲染，这显然十分浪费性能。

​ 但如果我们用 useCallBack 给 increment 进行包裹，并且第二个参数传入 count，则只有 count 改变的时候，increment 才会被重新定义，所以此时再修改 message 时，home 组件不会被重新渲染。

useMemo

作用

​ useMemo 的作用和 useCallback 大同小异，其作用有两个，第一是为了缓存值，第二是防止子组件多次渲染。而 useMemo 传入的函数是需要返回的是一个函数。在官网中，也有一句话对两个的区别进行了阐述。

useCallback(fn, deps)相当于useMemo(() => fn, deps)

案例

URL和URI的区别

发表于 2022-09-06 分类于 http

初始URL

​ URL全称为Uniform Resource Locator，代表着是统一资源定位符。用大白话来讲，URL正是使用Web浏览器等访问Web页面时需要输入的网页地址。

URI与URL的区别

​ URI全称为Uniform Resource Identifier，代表着是统一资源标识符。

​ URI用字符串标识某一互联网资源，而URL表示资源的地点（互联网上所处的位置）。可见URL是URI的子集。

​ 上面的概念看起来依旧晦涩难懂，我们拿人来做例子。假如世界上每个人的名字都不相同，那么名字就是URI其中的一个实例，我们可以通过每个人的名字来精确标识出唯一的人。

​ 但是现实中每个人的名字肯定是重复的，但是每个人的身份证号是肯定不同的，所以我们可以通过身份证号来精确标识出每一个人，这里身份证号就是URI的一个实例。

​ 那么URL在这个例子中可以如何体现呢？我们可以对比HTTP中URL的格式，就可以有：

​ 人类居住协议://地球/中国/湖北省/武汉市/某大学/某宿舍/xxx号寝室/张三

​ 我们可以通过这个特殊的URL，来精确的标识出某一个人。而这个URL，因为也起到了URI的作用，所以也是URI的一个实例。

​ 在这个例子中，我们既可以通过身份证号来精确标识出某一个人，也可以通过具体的居住信息来标识出某一个人，这些都是URI作用的体现，而URL的作用体现则是体现在后者的居住信息上。所以说URI是URL的一个子集。

​ 回到Web上，假如每个html文档都有自己唯一的编号，那么这个编号就是URI，则我们通过访问这个编号来访问具体的文档内容。而URL则是我们通过哪台主机哪个路径来去确定具体的资源，这种是通过路径定位的方式来确定具体的URI。

​ 用一句话总结来说，URL只能表示某一互联网资源（URI）的一个属性，即它的位置，而这个资源或许有其他的属性。

Loader && Plugin

我们首先看下在官方文档中，对于这两个东西的描述：

While loaders are used to transform certain types of modules, plugins can be leveraged to perform a wider range of tasks like bundle optimization, asset management and injection of environment variables.

​ 上面的话翻译过来就是：

  • Loader是用于特定的模块类型进行转换
  • Plugin可以用于执行更加广泛的任务，比如打包优化、资源管理、环境变量注入等；

loader是文件的加载器，它能够加载资源文件，并对这些文件进行一系列的处理，如编译、压缩等。例如将.less文件转换为.css文件。

而对于plugin，在webpack运行的生命周期会广播出很多的事件，plugin会监听这些事件，并在合适的时机通过webpak的api来改变输出结果。

深入理解BFC

BFC的定义

​ BFC全称为Block Formatting Context，直译为”块级格式化上下文“。它是一个独立的渲染区域。只有Block-level box参与，它规定了内部的Block-level box如何布局，并且与这个区域外部毫不相干。

block-level box即块级元素

BFC的生成

在MDN中，有总结BFC的生成方法：

  • 根元素（）

  • 浮动元素（元素的 float 不是 none）

  • 绝对定位元素（元素的 position 为 absolute 或 fixed）

  • 行内块元素（元素的 display 为 inline-block）

  • 表格单元格（元素的 display 为 table-cell，HTML表格单元格默认为该值），表格标题（元素的 display 为 table-caption，HTML表格标题默认为该值）

  • 匿名表格单元格元素（元素的 display 为 table、table-row、 table-row-group、table-header-group、table-footer-group（分别是HTML table、
    row、tbody、thead、tfoot 的默认属性）或 inline-table）

  • overflow 计算值(Computed)不为 visible 的块元素

  • 弹性元素（display 为 flex 或 inline-flex 元素的直接子元素）

  • 网格元素（display 为 grid 或 inline-grid 元素的直接子元素）

  • display 值为 flow-root 的元素

BFC的约束规则

  1. 在BFC中，box会在垂直方向上一个挨着一个排布；
  2. 垂直方向上的间距由margin属性决定；
  3. 在同一个BFC中，相邻两个box之间的margin会被折叠
  4. 在BFC中，每个元素的左边缘是紧挨着包含块的左边缘

BFC的作用

清除外边距margin折叠问题

​ 首先我们看个案例：

​ 我们分别给两个box设置了对应的margin，理论上两个box的间距应该是box1的margin-bottom的100px加上box2的margin-top的50px，一共为150px，可实际上由于这两个box处在同一个BFC中，造成了margin折叠，所以两个盒子之间的margin为较大的那个值，即100px。要解决margin折叠，我们可以让两个盒子处在两个BFC中，这样就可以解决margin折叠问题。

​ 我们可以给box2添加一个wrap，然后通过overflow:hidden触发这个box2的BFC，让box1和box2处于两个不同的BFC中，这样就可以解决外边距margin的折叠问题。

​ 可能有人会问，为啥不直接给box2设置overflow:hidden属性，这样box1和box2不也是处在两个不同的BFC了吗？答案肯定是不行的，前面我们说过，html根元素本身就会触发BFC，如果我们单纯的给box2设置，box1和box2依旧处在同一个BFC（根元素触发的BFC）中，而我们嵌套了一层后，box2处于的BFC是wrap创造的，box1处于的BFC是html根元素创造的。

函数模块化

发表于 2022-08-29 更新于 2022-08-30 分类于 JavaScript

什么是模块化

​ 首先来看维基百科对模块化的定义：

​ 模块化编程（英语：modular programming），是强调将计算机程序的功能分离成独立的、可相互改变的模块（module）的软件设计软件设计)技术，它使得每个模块都包含着执行预期功能的一个唯一方面（aspect）所必需的所有东西。

​ 通俗易懂的来讲，我们所说的模块化开发，就是将一个程序划分成一个个小的结构，按照这个结构划分去开发程序就是所谓的模块化开发。

​ 在每个结构结构中，我们可以编写对应的逻辑代码，而且这些逻辑代码在这个结构中有独有的作用域， 在这个结构可以将自己希望暴露的变量、函数、对象等导出给其结构使用；也可以通过某种方式，导入另外结构中的变量、函数、对象等。

模块化的历史背景

​ 摘自MDN：

​ JavaScript 程序本来很小——在早期，它们大多被用来执行独立的脚本任务，在你的 web 页面需要的地方提供一定交互，所以一般不需要多大的脚本。过了几年，我们现在有了运行大量 JavaScript 脚本的复杂程序，还有一些被用在其他环境（例如 Node.js）。

​ 因此，近年来，有必要开始考虑提供一种将 JavaScript 程序拆分为可按需导入的单独模块的机制。Node.js 已经提供这个能力很长时间了，还有很多的 JavaScript 库和框架已经开始了模块的使用（例如，CommonJS 和基于 AMD 的其他模块系统 如 RequireJS， 以及最新的 Webpack 和 Babel）。

​ 所以由此我们可以看出，无论你多么喜欢JavaScript这门语言，没有模块化是早期存在的极大痛点之一，虽然社区中给出了很多方案，但是JavaScript本身，直到ES6（2015）才推出了自己的模块化方案。而在此之前，为了让JavaScript支持模块化，涌现出了很多不同的模块化规范：AMD、CMD、CommonJS等

早期没有模块化的问题及解决方案

​ 没有模块化最大的问题之一就是命名冲突。

​ 假如甲乙在编写同一个程序，然后他们在各自实现对应的子功能的js文件里使用相同的变量名，最后在合并狼个人的代码时就会出现命名冲突的问题。

​ 在早期的解决方案中，我们可以用**IIFE(立即执行函数)**来解决，将逻辑代码用立即执行函数包裹成一个模块，将这个模块进行导出

​ 但是该方案看起来显然不优雅，首先使用立即执行函数包裹代码会显得代码看起来混乱，同时没有合适的规则情况下，模块的命名也会造成混乱。

​ 在计算机科学中，柯里化（英語：Currying），又译为卡瑞化或加里化，是把接受多个参数的函数变换成接受一个单一参数（最初函数的第一个参数）的函数，并且返回接受余下的参数而且返回结果的新函数的技术。

​ 柯里化声称，如果你固定某些参数，你将得到接受余下参数的一个函数。

​ 上面的解释是非常抽象的，我们用大白话来解释一下，所谓的柯里化，就是只传递函数一部分参数给一个函数，然后返回这个函数，利用这个返回的函数去处理原函数的剩余参数。

​ 这个过程就叫做柯里化。
